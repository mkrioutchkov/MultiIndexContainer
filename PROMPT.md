# Greenfield C++23 Multi-Index Container â€” Engineering Specification

**Mission.** Build a fully-fledged, greenfield, value-semantic multi-index container for the latest MSVC `/std:c++latest` toolchain (clang-cl/clang/gcc parity as a hard goal) that achieves complete behavioral parity with `boost::multi_index_container` â€” every index family (ordered, hashed, sequenced, ranked, random-access), uniqueness constraints, key extractors, composite keys, tags, projection, node-handle transfer, the full lookup/modify surface, and the documented complexity/exception/invalidation guarantees â€” and then deliberately surpasses Boost by exploiting modern C++23/26 facilities (concepts, ranges, deducing-this, `std::expected`, modules, `std::format`, pmr, flat storage, string-literal tags, coroutines) to deliver dramatically better diagnostics, ergonomics, safety, and introspection. Boost is the executable oracle; "everything Boost does, plus the greenfield extras Boost can never retrofit" is the bar.

---

## Goals & Non-Goals

**Goals**
- Full feature parity with `boost::multi_index_container<Value, indexed_by<...>, Allocator>`: the single-node-per-element model (one physical copy threaded into every index), all index kinds, all extractors, composite keys, projection, node handles, and the published complexity/exception/invalidation signatures.
- A modern, declarative, type-list-driven API (`multi_index<Value, indexed_by<...>>`) with concept-constrained customization points and first-class, human-readable diagnostics.
- Strict per-operation complexity, exception-safety, and iterator/reference-stability contracts, all tested against the Boost oracle.
- Zero-overhead single-threaded default; opt-in extras (observers, stats, concurrency, flat storage, pmr) that vanish entirely when unused.
- Primary toolchain MSVC `/std:c++latest`; warning-clean parity on clang (+libc++ and +libstdc++), clang-cl, and gcc.

**Non-Goals**
- Runtime-pluggable / type-erased index topology (a "document store"). Index sets are fixed at compile time â€” explicitly out of scope; direct such users to a separate product.
- A frozen binary ABI across versions (template/header/module distribution makes this infeasible; see Versioning).
- Dependency on Boost in shipped artifacts (Boost appears only as a test-time oracle).
- Hard dependency on static reflection (P2996) â€” architect for it, do not require it (no MSVC support as of mid-2026).

## Required Index Types

Each is a distinct engine with its own iterator category, lookup surface, and invalidation rules. All index iterators are **constant** (mutation only via `modify`/`replace`/`extract`).

- **`ordered_unique` / `ordered_non_unique`** â€” balanced BST (red-black). Bidirectional const iterators; ascending key order; `ordered_non_unique` permits duplicates and preserves insertion order among equivalents (stable). Insert `O(log n)`; hinted insert amortized `O(1)`; erase amortized `O(1)`; `find/lower_bound/upper_bound/equal_range/contains` `O(log n)`; `count` `O(log n + matches)`. Heterogeneous ("compatible-key") lookup with optional compatible comparator.
- **`hashed_unique` / `hashed_non_unique`** â€” bucketed hash table. **Forward** const iterators + per-bucket forward local iterators; unspecified iteration order; `hashed_non_unique` groups equivalents contiguously (`equal_range` `O(1)` avg). Average `O(1)` insert/erase/find/replace/modify, worst `O(ndist)`. **Rehash does NOT invalidate iterators/pointers/references** (only relative order may change) â€” a deliberate divergence from `std::unordered_*`. Full bucket interface: `bucket_count`, `max_bucket_count`, `bucket_size(n)`, `bucket(k)`, `begin(n)/end(n)/cbegin(n)/cend(n)`, `local_iterator_to`, `load_factor`, `max_load_factor[(z)]`, `rehash(n)` (strong guarantee), `reserve(n)`. No ordering operations.
- **`sequenced`** â€” `std::list`-like free-order bidirectional sequence (insertion order by default, no key). `O(1)` insert/erase/relocate anywhere. Full list API: `front/back`, `push_front/push_back` (return `pair<iterator,bool>`; `false` if another index's uniqueness rejects it), `pop_*`, `emplace_*`, `assign`, positional `insert`, `erase`, `splice` (cross-container transfer; rejected elements remain in source), `relocate` (in-place reposition, `O(1)`/`O(distance)`), `remove/remove_if`, `unique[(pred)]`, `merge[(comp)]` (both sorted, stable), `sort[(comp)]` (stable), `reverse`. No key lookup.
- **`random_access`** â€” free-order sequence with **random-access** const iterators and array-like positional access over an internal pointer array. `operator[]`, `at` (throws `std::out_of_range`), `front/back`, `nth(n)`, `index_of(it)`, `capacity/reserve/resize/max_size`, plus the full list-op suite (`rearrange` included). Amortized `O(1)` end-insert; positional insert/erase `O(distance-to-end)`; access/replace `O(1)`. **Pointer-array reallocation does NOT invalidate iterators/pointers/references to elements** (nodes are stable) â€” diverges from `std::vector`. No `push_front` (use `insert(begin(), x)`).
- **`ranked_unique` / `ranked_non_unique`** â€” ordered index augmented with subtree-size bookkeeping (order-statistics tree). Same ordered interface and semantics, **plus** `nth(n)`, `rank(it)`, `find_rank(x)`, `lower_bound_rank(x)`, `upper_bound_rank(x)`, `equal_range_rank(x)`, `range_rank(lower, upper)` (all `O(log n)`). Trade-off vs ordered: hinted insert and erase become `O(log n)` (not amortized `O(1)`); higher per-node overhead.

> The container requires at least one index; an empty `indexed_by<>` is ill-formed. Container-level `begin/end/size/empty/...` forward to the first declared index. A multi-index container may itself be a `Value` of another (nesting), subject to the per-index key requirements.

## Key Extraction & Composite Keys

- **Identity / member extractors:** `identity<T>`; `member<Class, Type, &Class::field>` (member-pointer NTTP). Extractors apply through pointer chains and `reference_wrapper` (dereference until reaching `Type&`). `result_type` is the key type; equal extractors on equal objects yield equal keys; extractors are stateless/default-constructible by default.
- **Computed-key extractors:** `const_mem_fun` / `mem_fun` and the full CV/ref-qualified family (`cv_`, `cref_`, `cvref_`, `volatile_`, `ref_`, `vref_`); `global_fun<Value, Type, &f>` for free/static functions (key a type you cannot modify). Member-function keys are computed on demand, not stored.
- **`key<>` shorthand (C++17+ NTTP deduction):** `key<&Class::member>` â†’ member extractor; `key<&Class::method>` â†’ `(const_)mem_fun`; `key<&freefn>` â†’ `global_fun`; `key<&A::x, &B::y, ...>` â†’ composite key, deducing the least-generic value type automatically.
- **Composite keys:** `composite_key<Value, K0, ..., Kn>` producing a tuple-like key compared **lexicographically**, with `composite_key_compare<...>` (ordered/ranked, strict-weak-ordering, `std::less`/`std::greater` specializations) and `composite_key_hash<...>` + `composite_key_equal_to<...>` (hashed; consistent: equivalent keys must hash equal). Per-component comparator/hash/pred overridable.
- **Partial / prefix lookups (the headline composite-key feature):** a tuple of the first *k* components (`k â‰¤ n`) is a compatible key, enabling `lower_bound/upper_bound/equal_range/find` on a prefix in ordered/ranked indices, and `find/equal_range` on full-or-prefix tuples in hashed indices (when consistent). Prefix field order must match the key's declared order (compile-time enforced). Hashed composite keys support **full-key only** â€” the type system must not expose prefix/range methods on hashed indices.
- **Modern key ergonomics (extras):**
  - **Aggregate composite keys:** allow a plain `struct` key (`struct name_age { std::string name; int age; auto operator<=>(const name_age&) const = default; };`) as an alternative to `composite_key<>`. Build lookup keys with designated initializers (`find({.name="x", .age=42})`), decompose results with structured bindings, and **auto-derive the comparator from `operator<=>`** (`three_way_comparable` detection) so no explicit `Compare` is needed for the common lexicographic case. Provide `prefix(...)`, `key_ge(...)`, `between(low, high)` builders for partial/open composite ranges that read clearly while preserving `O(log n)` lexicographic semantics.
  - **Transparent by default:** ordered comparators default to transparent (`is_transparent`); hashed indices require a transparent hash+equality pair. `find("literal")` / `equal_range(string_view)` perform **zero key materialization**. Non-transparent functors are a clearly-diagnosed fallback (or hard error), never a silent slow path. Debug mode audits that transparent and homogeneous comparisons agree on sampled inserts.
  - Keep the extractor layer behind a small trait so a future reflection backend (`index<by<"name">>`) can be dropped in without breaking the member-pointer/lambda API shape.

## Tagging Model

> **DECISION FORK â€” confirm before implementation. Recommended default: ENFORCE mandatory unique tags; by-tag access only; no by-number `get<N>()`.**

- **Recommended (enforced tags):** every index MUST carry at least one unique tag; access is `c.index<by_id>()` / `c.get<by_id>()`, never `c.get<0>()`. Rationale: by-number access is the single biggest source of silent breakage in Boost (inserting/reordering an index renumbers every call site with no compiler help; `get<2>()` is unreadable). Enforced tags make access self-documenting, make reordering a no-op for callers, and let diagnostics name the index (`by_email`) instead of a position. Mitigations for the ceremony cost: a `default_tag` for the single-index degenerate case, and `ordered_index<...>`/`unique_index<...>` convenience aliases that auto-generate a tag. This is a deliberate, documented, breaking divergence from Boost.
- **Boost-compatible alternative (if the fork is rejected):** retain optional tags plus numeric `get<N>()` and container-level forwarding to index 0, exactly as Boost.
- **Common to both:** multiple tag aliases per index allowed (`tag<a, b, c>`); tags unique across the whole container (compile-time error on collision, naming both colliding indices and the duplicated tag); "tag not found" / "index out of range" are compile-time errors.
- **Tag representation â€” secondary fork (recommended: support both):** offer **string-literal NTTP tags** via a structural `fixed_string` (`get<"by_name">()`, `index<"by_name">`) for readability, formattability, and typo-checking, **and** classic empty-struct type tags for ADL-friendly distinct types. String tags make error messages and `{}`-dumps print the actual tag name.
- Metaprogramming surface: `index<Container, Tag>::type`, `index_iterator<...>::type`, const variants â€” the named handles `project<>`/`iterator_to` rely on.

## Core Operations & Semantics

- **Construction:** default; from `ctor_args_list` (per-index comparator/hash/pred instances) + allocator; range `[first, last)` (+ optional args/allocator); `initializer_list`; copy; move; allocator-extended variants; **ranges-aware** `from_range_t`. Copy = `O(n log n)` (rebuild every index); move = `O(1)` (steal nodes); move-assign `O(1)` when allocators compatible else element-wise. Self-assign/self-swap are safe no-ops.
- **Insertion & the all-indices uniqueness rule:** `insert(x)` / `insert(x&&)` / `emplace(args...)` return `pair<iterator, bool>` (`true` iff inserted; on failure `first` points to the blocking element). Hinted forms return an `iterator`. Range/`initializer_list` inserts return `void`, skipping rejected elements. **Insertion succeeds only if it satisfies the uniqueness of EVERY unique index simultaneously**; rejection by any one aborts the whole insertion. A temporary may be constructed then destroyed to test the key (emplace).
- **Erasure & node handles:** `erase(iterator)` â†’ next iterator (nothrow); `erase(key)` â†’ count (basic guarantee); `erase(first, last)` â†’ `last` (nothrow); `clear()` `O(n)` nothrow. Node handles: `extract(const_iterator)` / `extract(key)` detach a node from **all** indices without deallocating, yielding a movable `node_type` that owns the value+allocator and exposes mutable `value()` (the only zero-copy way to re-key); `insert(node_type&&)` re-links across all indices returning `insert_return_type{position, inserted, node}` (node retained on failure); hinted `insert(pos, node_type&&)` â†’ iterator. While extracted, pointers/refs dangle but **revalidate on successful re-insertion** (iterators do not). Cross-container transfer requires matching index layout (enforced via the handle's type) and compatible allocator; cross-allocator falls back to move-construct + insert with a clear diagnostic. `merge(other)` / `merge(other.index<tag>())` splices all extractable nodes, leaving uniqueness-rejected ones in the source.
- **In-place mutation (`replace`/`modify`/`modify_key`) â€” the subtlest surface:**
  - `replace(pos, x)` â†’ `bool`; assigns if all constraints still hold (else no change, `false`); position stays valid; **strong** guarantee.
  - `modify(pos, mod)` invokes `mod(elem)`, then re-positions in every index; `â†’ bool`. Modifying a key that only violates **ordering** is fine (auto-repositioned). Modifying a key that violates a **unique** constraint, or `mod` throwing, triggers rollback.
  - `modify(pos, mod, rollback)` supplies an explicit inverse. **Recommended default semantics (divergence from Boost â€” confirm):** rollback-and-**keep** the element in its original position (snapshot only the keys of indices whose key actually changed); the legacy Boost erase-on-failure behavior is available behind an explicit `on_collision::erase` policy for migrators. (Document precisely which is the default; this is the highest-value safety improvement and must be exact for drop-in compatibility.)
  - `modify_key(pos, mod[, rollback])` passes only the key (requires a writable extractor, e.g. `member`, not `const_mem_fun`).
- **Lookup:** ordered/ranked â€” `find/count/lower_bound/upper_bound/equal_range/contains` + `range(lower_pred, upper_pred)` with `unbounded` sentinels; all accept compatible keys + optional compatible comparator (precondition: compatible comparison partitions the index identically). Hashed â€” `find/count/equal_range/contains` + `(CompatibleHash, CompatiblePred)` overloads (no `lower/upper_bound/range`; compatible keys must hash equal). Complexity as listed per index above.
- **Projection & iterator interconversion:** `project<to_tag>(it)` (and method form `it.project<to_tag>()`) converts to the same-element iterator in another index in `O(1)`; `index<to_tag>().iterator_to(value_ref)` and hashed `local_iterator_to`. `project(end())` â†’ target `end()`. Source/target must be indices of the **same** container instance; `project<to_tag>` fails to compile if `to_tag` is not a tag of this container (diagnostic lists available tags); const-correctness preserved; cross-container iterators are a debug-caught precondition violation.
- **Comparison & swap:** ordered/ranked indices define `== != < <= >= >` (lexicographical); hashed define `== !=` only (order-independent multiset equality). Container-level `==` uses index 0's semantics. `swap` is `O(1)`, nothrow when comparators/hashers/allocators swap without throwing, re-associates iterators to the swapped container while keeping them valid.

## Exception Safety, Invalidation & Complexity Guarantees

- **Strong guarantee** for single-element `insert`/`emplace`, `erase`, node `extract`/`insert`, `replace`, and (per the recommended fork) `modify`/`modify_key` with rollback: on any throw â€” element construction, key extraction, comparison, hashing, allocation â€” the container is exactly as before and all iterators stay valid. **Basic guarantee** (documented) for range/bulk insert; opt-in `insert_atomic(range)` gives all-or-nothing via a staging buffer.
- **`noexcept` accuracy:** conditional `noexcept` derived from element ops, comparators, hashers, and allocator traits â€” never blanket `noexcept` that could mask a throwing user functor. No error codes or exceptions-as-control-flow internally; allocation failure propagates the allocator's exception.
- **Invalidation contract (a defining advantage over std containers):** iterators (including `end()`), pointers, and references stay valid for the container's lifetime and are invalidated **only** when their element is erased/extracted. Extracted-node pointers/refs revalidate on re-insertion; iterators do not. **Hashed rehash and random-access reallocation invalidate nothing** â€” contrast with `std::unordered_*`/`std::vector`. `const_iterator` and `iterator` are effectively identical (both const); `rbegin/rend` on bidirectional/random-access; `cbegin/cend` everywhere.
- **Complexity signatures are a hard contract** and compose additively: a multi-index operation's cost is the **sum over all indices** it must update (state this explicitly for `insert` and for `modify`, the latter being the cost of repositioning in the modified index plus the sum across all other indices). Per-index Big-O is exactly as enumerated above and must be met by the chosen data structures (RB tree, augmented RB tree, hash table, doubly-linked list, pointer array). No performance target may be met by sacrificing a documented complexity guarantee.
- **Memory guarantee:** exactly **one node allocation per element regardless of index count** (the intrusive-node advantage over composing N separate containers); per-node overhead stated as an exact formula in index count/types and asserted by a static test; all allocations route through the provided allocator (verified by a throwing-global-new test).

## Modern C++23 Enhancements (the "additional things on top")

- **Concepts at the index-spec boundary:** `key_extractor_for<E, Value>`, `strict_weak_order_for<Cmp, Key>`, `hasher_for<H, Key>`, `equality_for<P, Key>`, element-type concepts â€” so a bad extractor/comparator yields a one-line "constraints not satisfied" diagnostic at the `typedef` site, not a multi-screen backtrace. Concepts also disambiguate member-pointer vs callable vs (future) reflection-derived extractors.
- **`std::ranges` integration:** every index models at least `bidirectional_range` (random-access where applicable); ship lightweight, copyable **index view** types (`view` / `borrowed_range`, marked `enable_borrowed_range` when non-owning) so `mic.get<by_name>() | views::filter(...) | views::transform(...) | views::take(10)` composes directly. `std::ranges::equal_range(idx, key, cmp, proj)` returning a `subrange` subsumes much of Boost's `range()` ergonomics; `from_range_t` construction and `std::ranges::to<...>` round-trips supported. Filtered/transformed views over a mutating container must respect node stability.
- **Deducing this:** `template<class Self> auto&& get(this Self&& self)` for `get<Tag>`, `operator[]`, `front/back`, and index accessors â€” one body yields all cvref overloads, halving the accessor surface, guaranteeing const-correctness, enabling move-out from rvalue containers, and recovering the derived index type without CRTP. Keep accessor bodies simple (known recursive-lambda/overload-set interactions).
- **`[[no_unique_address]]` (with `[[msvc::no_unique_address]]`):** store stateless extractors/comparators/hashers/allocators as zero-byte members instead of EBO base soup. **MSVC caveat is load-bearing:** the plain attribute is a no-op on MSVC for ABI reasons â€” apply both spellings via a macro so the layout optimization actually happens on cl.exe. Treat the chosen spelling as ABI-affecting.
- **`constexpr`/`consteval` containers & compile-time tests:** make construction/insert/find/iterate/erase `constexpr` for the realistic index families (ordered/sequenced/random-access; hashed likely excluded â€” be honest), enabling `consteval make_table(...)` immutable lookup tables and `static_assert`-based invariant tests ("all indices agree on size", "erase removes from all indices", "composite ordering consistent") that run at compile time. Use `if consteval` to select a constexpr-friendly flat representation during constant evaluation and a pointer-node/SIMD-friendly layout at runtime, with **identical observable behavior** â€” only representation/performance differ.
- **`std::expected` fallible API (alongside classic returns):** error-returning overloads `expected<iterator, insert_error>` and `expected<iterator, modify_error>` where the error names the offending index/reason (duplicate in unique index N, comparator threw, capacity exceeded). Keep `pair<iterator, bool>` for source compatibility. This is where the non-destructive `modify`/`replace` story shines â€” failure is explicit and the element survives. Monadic `and_then`/`transform`/`or_else` compose.
- **Coroutine query helpers:** a `queries` namespace of `std::generator<const T&>`-returning free functions â€” `merge_join(indexA, indexB, on_key)`, `group_by(index, keyproj)`, streaming `equal_range` â€” that read like SQL while staying lazy and `O(1)` space. Document the heap-frame/indirect-call overhead and always offer an eager ranges path for hot loops; generators are ergonomics, not the fast path.
- **`operator<=>`-derived comparators & aggregate/structured composite keys:** as in Key Extraction â€” `three_way_comparable` keys synthesize the comparator; designated-initializer construction and structured-binding decomposition replace the `composite_key_result` proxy where the user opts into a struct key. Surface the resulting ordering strength so unique-vs-multi semantics stay correct.
- **Flat / vector-backed storage policy:** an `ordered_unique<..., storage::flat>` family (Ă  la `std::flat_map`/`std::flat_multiset`) storing elements once in a contiguous arena with each ordered index as a sorted vector of handles â€” trading insert/erase cost for cache locality, smaller memory, and constexpr-friendliness. **Stability differs sharply by policy** (flat handles invalidate on reallocation) â€” make the per-policy iterator/handle invalidation contract explicit. `storage::node` remains the default classic engine.
- **`std::pmr` support:** route node/arena allocation through `std::pmr::polymorphic_allocator` (textbook fit for the many same-sized nodes â€” `unsynchronized_pool_resource`, `monotonic_buffer_resource`); a `pmr::multi_index` alias and `memory_resource*` constructors; type-erased allocator state so different-resource containers share a type.
- **`std::inplace_vector` static variant:** a `static_multi_index<T, N, Indices...>` backed by `std::inplace_vector<T, N>` (or a vendored shim) â€” bounded capacity, all-inline, zero heap, ideal for embedded/real-time/constexpr and a natural pair with flat storage. Over-capacity insert is exactly the `expected(unexpected(capacity_exceeded))` case. **Gate on the `inplace_vector` feature-test macro (C++26, may not yet ship in your MSVC) and ship a fallback.** Do not design around `std::hive` (did not make C++26).
- **`std::move_only_function`** for runtime/type-erased callable parameters (visitors, runtime comparators, batch callbacks) so move-only state is allowed â€” but keep compile-time functor template params (with `[[msvc::no_unique_address]]`) for EBO/inlining; do not replace them.
- **`std::span`/`std::byte`/`std::start_lifetime_as`** for bounds-safe bulk APIs (`insert(std::span<const T>)`) and standards-conformant arena lifetime management (no `reinterpret_cast` UB); `start_lifetime_as` for trivially-constructible scaffolding, real placement-new for elements.
- **`[[assume]]` now, C++26 contracts later:** encode invariants ("all indices equal size", "node linked in every index", "comparator consistent") as debug-validated checks, feeding `[[assume(expr)]]` on hot paths after validation. Never put a side-effecting or possibly-false expression in `[[assume]]`. C++26 `pre`/`post` contracts are a future upgrade (no MSVC support mid-2026).
- **Static reflection (P2996) â€” design for, do not depend on:** put the extractor layer behind a trait so a reflection backend (member-name tags `index<by<"name">>`, auto `operator<=>`, auto `std::formatter` over members, tag-vs-member validation) can drop in later. **No MSVC support, no ETA as of mid-2026** â€” ship member-pointer/lambda extractors today; keep reflection off the critical path.

## API & Ergonomics

- **Declarative type-list is canonical:** `multi_index<Value, indexed_by<ordered_unique<...>, hashed_non_unique<...>, sequenced<...>, ranked_unique<...>, random_access<...>>>`. `Value` stated once as the first parameter, never repeated per index. **No fluent builder for index topology** (would imply type erasure / runtime dispatch, defeating zero-overhead node layout). A small fluent/config surface exists only for non-topological knobs: `.reserve()`, `.max_load_factor()`, `.rehash()`, and designated-initializer construction config (`multi_index c{ .reserve = 1024, .observers = ... }`).
- **Curated compile-time validation suite** (each check its own concept, gating the heavy body so the first failure fires a short message): (1) at least one index â€” "multi_index requires at least one index; indexed_by<> is empty"; (2) unique tags â€” names both colliding indices and the tag; (3) extractor invocable on `const Value&` and its result usable by the paired comparator/hasher â€” "comparator std::less<int> is not callable with key type 'std::string' produced by extractor member<Rec,&Rec::name>"; (4) hashed hasher+equality both accept the key; (5) member-pointer extractors structurally belong to `Value`; (6) allocator rebinds to the node type. Provide `MULTI_INDEX_DIAGNOSTIC_VERBOSE` to dump the computed node layout and per-index trait table.
- **Statistics & introspection (`collect_stats` policy for the costly bits):** per-index, kind-appropriate stats â€” ordered/ranked sizes (and rank/select); hashed `bucket_count`, `load_factor`, `max_load_factor`, longest probe/chain, collision count, `hash_quality()` (spread metric to diagnose a bad hasher). `c.stats()` returns a structured, formattable aggregate; `c.audit()/introspect()` returns per-index counts (all must be equal â€” a cheap corruption probe) and memory accounting (node size, per-index overhead, allocator high-water mark). Cheap derivable stats (sizes, load factor) are always on; running counters are behind the policy.
- **Debug-mode invariant auditing (`debug_policy` tiers: off / on_demand / per_mutation):** `audit_invariants()` checks every index holds the same node set (equal size + cross-index membership), ordered indices sorted per comparator, hashed nodes in their correct bucket, ranked subtree counts correct, sequenced/random-access links form a valid cycle. On failure, abort with a formatted dump of the offending node/index. `per_mutation` is `O(n)`/op, test-only; release builds unaffected.
- **Opt-in observers/validators (`observers<...>` policy or `.subscribe(...)` RAII token):** post-commit, cannot-fail hooks (`on_insert`, `on_erase`, `on_before/after_modify`, `on_clear`) that always see a consistent container, plus a separate pre-commit validator category that may **veto** insert/modify (participating in rollback). Type-erased by default with a zero-overhead templated form for hot paths; the entire facility and its per-node storage vanish (`[[no_unique_address]]`) when no `observers<>` policy is present.

## Concurrency

> **DECISION FORK â€” confirm the model. Recommended default: three explicit tiers chosen at the type level; never a "thread-safe" switch on the default type.**

1. **`multi_index<...>` (default):** single-threaded, no internal synchronization, zero overhead â€” same contract as `std::` containers (concurrent reads OK; any write needs external exclusion).
2. **`externally_locked<...>` wrapper / strand policy:** bundles a `shared_mutex` and exposes RAII `read_handle`/`write_handle` so a caller can hold one consistent lock across a multi-index transaction (find in index A, then erase via the projected iterator in index B) â€” essential because cross-index operations must be atomic.
3. **`concurrent_multi_index<...>` (separate type):** fine-grained/sharded locking (per-bucket locks for hashed; optional optimistic reads) with a deliberately **narrower, closure-based API** â€” no raw iterators outliving a lock; lookups return values/handles; mutations are closures (`upsert`, `compute`, `visit`). Sharding is a policy knob.

Rationale: the hard part of multi-index concurrency is that one operation spans several indices, so the consistent unit of locking is the whole container or a coordinated set â€” naive per-index locks deadlock or tear. The concurrent variant cannot expose the full iterator API (iterators + concurrent mutation = invalidation hazards), so it is intentionally a different, narrower type rather than a flag on the main one. Do **not** make the default "thread-safe with a switch" (slow for everyone, still subtly wrong). TSan is the executable check on the documented read-concurrent contract.

## Serialization & Formatting

- **Pluggable serialization via a thin `Archive` concept (NOT tied to Boost.Serialization):** `save(ar, value)` / `load(ar, value)` free-function CPOs plus container-level `save_container` / `load_container`. Persist **one index's element sequence plus metadata**, then **rebuild all other indices on load** (compact, version-robust, survives index-layout changes) â€” costing `O(n log n)` rebuild vs persisting fragile internal pointers. Optional fast-path: persist sequenced/random-access order to preserve insertion order exactly. Ship adapters for a built-in binary archive and `std::ostream/istream` text, plus trait hooks so users can wire Cereal/Protobuf/Boost.Serialization without the library depending on any of them. Load-time uniqueness violations reported with the offending element via a load policy (throw vs skip-and-collect). Debug-mode round-trip self-check (serialize â†’ deserialize â†’ `audit_invariants`). Optionally reproduce Boost's distinctive **serializable iterators** (an iterator saved and reloaded points at the corresponding element) or explicitly scope it out.
- **`std::format` / `std::formatter` (first-class, not ad-hoc `operator<<`):** the container and each index model `std::formatter`. Mini-language: `{}` â†’ summary (type + per-index sizes); `{:full}` â†’ every element via its own formatter; `{:index=tag}` â†’ dump in that index's order; `{:stats}` â†’ the statistics report; `{:audit}` â†’ the invariant-audit result. `dump(std::ostream&)` and `to_string()` wrap the formatter; integrates with `std::print`/`std::println`. Element-level dumps require `Value` formattable (checked at the call site, not on the type); summary/stats work regardless. Format the error types (`insert_error`/`modify_error`) and tags too. Stats/audit format paths reuse the structured stats objects â€” one source of truth for programmatic and printed introspection.

## Testing & Verification

- **Unit tests** (Catch2 v3 or doctest â€” pick one): every public API of every index and every composition â€” all constructors (default/copy/move/range/`initializer_list`/allocator-extended), iterator/const/reverse on each index, projection, node-handle extract/insert, comparison operators, swap, all modifiers, exception paths. - **Property-based tests** (RapidCheck) â€” oracle-independent invariants: (1) every index reflects the same element multiset; (2) ordered indices always sorted per extractor+comparator; (3) hashed bucket-consistency; (4) sequenced/random-access order semantics; (5) `size()` equals true count across indices; (6) insert-then-erase round-trips to prior observable state; (7) documented iterator stability across modify/insert/erase. â‰Ą1000 cases/property per PR, 100k nightly, with shrunk historical failures replayed every build (persist failing seeds as fixtures).
- **Differential testing against `boost::multi_index_container` (critical deliverable, the executable spec):** drive an identical randomized op stream against our container and a Boost container with the **same** index spec; assert observable equivalence after **every** op â€” per-index iteration order element-for-element, `find/lower_bound/upper_bound/equal_range/count`, `size/empty`, hashed bucket layout where Boost guarantees it, the `insert` `pair` return, the `erase`/`modify` iterator, and `modify`/`modify_key` success-or-rollback outcome. Cover the cross product of `{ordered_unique, ordered_non_unique, hashed_unique, hashed_non_unique, sequenced, random_access}` with 2â€“4 indices. Op alphabet: insert, emplace, `erase(key)`, `erase(iterator)`, modify (mutate each key), modify-success and rollback-on-collision, clear, swap, replace, splice, random-access reordering. **1M+ ops/seed nightly**, smoke subset per-PR; deterministic and seed-replayable; pin and record the Boost oracle version. **Equal-element ordering in non-unique indices is the highest-risk divergence** â€” match it or encode each intentional divergence as a harness exception with written rationale (cross-linked to the migration guide).
- **Fuzzing** (libFuzzer): decode bytes â†’ op sequence against a representative config, and a **second target against the differential oracle** (finds divergences, not just crashes). - **Exception-injection harness** (critical): a functor/allocator that throws on the Nth call, asserting strong-guarantee rollback and intact invariants at **every** throwing site, integrated with the differential oracle so rollback behavior is also checked against Boost.
- **Concept archetypes & diagnostics-quality bar:** instantiate the full container against minimal archetypes (move-only comparator, by-value vs by-reference extractor, move-only element, non-default-constructible element, throws-on-copy element) to prove the container over-constrains nothing; plus negative/SFINAE tests proving concept-failing types are rejected cleanly. **Diagnostics-snapshot suite:** intentionally ill-formed usages compiled with expected-error matching (grep the compiler output for a required human-readable substring on MSVC/clang/gcc), failing CI if message quality regresses. ## Benchmarking & Performance Targets

- **Methodology** (Google Benchmark): each primitive op in isolation at `N = 2^10, 2^14, 2^18, 2^22` and key distributions (sequential, uniform-random, zipfian, **adversarial-collision** for hashed). - **Targets** (measured by the above): (a) **vs Boost** â€” within **Â±10%** on insert/find/modify/iterate/erase across all index types and distributions, meet-or-beat on iteration, no single benchmark worse than **+15%**; (b) **vs std** â€” single-index ordered within **Â±15%** of `std::map`, single-index hashed within **Â±15%** of `std::unordered_map`, iteration within **Â±10%** of the std equivalent; (c) **memory-per-node** within **+1 pointer-width** of Boost's per-node overhead per index, stated as an exact formula and verified by the counting allocator; (d) **no target met by sacrificing a documented complexity guarantee.**

## Open Design Decisions to Confirm

1. **Tag enforcement (primary fork).** Recommended: **enforce mandatory unique tags, by-tag access only, no `get<N>()`** (a deliberate, documented, breaking improvement over Boost), with `default_tag` and auto-tagging convenience aliases for degenerate cases. Alternative: Boost-compatible optional tags + numeric access.
2. **Tag representation.** Recommended: **support both** string-literal `fixed_string` NTTP tags (`get<"by_name">()`) and classic empty-struct type tags.
3. **`modify`/`replace` failure semantics.** Recommended: **rollback-and-keep by default** (non-destructive, `std::expected`-friendly), with Boost's erase-on-failure available behind `on_collision::erase`. Must be documented exactly for drop-in compatibility.
4. **Fallible-API style.** Recommended: **offer both** `std::expected<iterator, â€¦>` overloads **and** classic `pair<iterator, bool>` / `bool`.
5. **Distribution model.** Recommended: **header-only primary + optional C++23 module + amalgamation**, all three gated through the same test suite.
6. **Concurrency model.** Recommended: **three explicit type-level tiers** (single-threaded default, `externally_locked` transaction wrapper, separate closure-based `concurrent_multi_index`) â€” never a switch on the default type.
7. **Storage policy default.** Recommended: **node-based default**, with an opt-in `storage::flat` policy (and per-policy invalidation contracts stated explicitly).
8. **Serializable iterators.** Confirm whether to reproduce Boost's serializable-iterator feature or explicitly scope it out.
9. **`std::inplace_vector` dependency.** Confirm the shipping MSVC build exposes it (C++26); otherwise enable the vendored shim behind a feature-test macro. Do not design around `std::hive` (not in C++26).
10. **Reflection posture.** Confirm reflection (P2996) stays **off the critical path** (architect the extractor trait for a future backend; ship member-pointer/lambda today â€” no MSVC support mid-2026).
11. **Testing-framework choice** (Catch2 v3 vs doctest) and **pinned compiler minimums** (MSVC toolset / clang / gcc majors) â€” confirm and record.
